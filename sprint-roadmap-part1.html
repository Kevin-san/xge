0. 整体规划总览

需求拆分方法论
本路线图采用「三层金字塔式需求分解」：
① 阶段（Phase）：4 个大阶段，每个阶段对应一个完整的里程碑交付
② Sprint（冲刺）：每个阶段拆分为 4 个 Sprint，两周一个 Sprint，总计 16 个核心 Sprint
③ 项目需求（PR / Project Requirement）：每个 Sprint 约 8~10 个项目需求，总计 130+
④ 细分需求（SR / Sub Requirement）：每个项目需求下再拆分为 10+ 个具体可验收的子任务，总计 1300+

0.2 优先级与验收标准说明

优先级含义适用场景验收标准
P0 · 阻塞级必须在本 Sprint 内完成，否则下一阶段无法启动核心引擎循环、基础渲染管线、关键数据结构通过单元测试 + 集成测试 + Demo 验证
P1 · 核心级本阶段核心功能，影响主要场景使用编辑器核心面板、物理引擎关键特性通过功能测试 + 性能基准测试
P2 · 增强级锦上添花功能，可延后至下一阶段高级特效、边缘平台适配、工具链增强通过功能测试即可

0.3 命名规范

  • 阶段编号：P1 / P2 / P3 / P4（Phase 1~4）
  • Sprint 编号：S1 / S2 / ... / S16
  • 项目需求 ID：PR-{Sprint编号}-{两位序号}，例如 PR-01-03 表示 Sprint 1 的第 3 个项目需求
  • 细分需求 ID：SR-{Sprint编号}-{PR序号}-{两位序号}，例如 SR-01-03-07 表示 Sprint 1 第 3 个 PR 的第 7 个子需求
